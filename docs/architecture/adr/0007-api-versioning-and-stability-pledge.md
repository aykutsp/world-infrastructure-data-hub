# ADR-0007 — API versioning and stability pledge

- **Status**: accepted
- **Date**: 2026-04-05

## Context

People (and scripts, and other apps, and dashboards, and AI agents) will build on top of the JSON endpoints. The instant we ship a breaking change without warning, every one of them breaks silently.

The dataset is also published under an open licence, so consumers have every right to mirror, vendor, or hot-link to the endpoints. We cannot retroactively control what they do with the bytes — we can only make promises about what the bytes will look like.

## Decision

The public JSON endpoints live under `/api/v1/`. That `v1` is a load-bearing commitment with three concrete promises:

1. **No removed fields, ever, within a major version.**
   Once a field is in the schema, it stays. If we want to replace it, we add the new field and deprecate the old one. The old one disappears only in `/api/v2/`.

2. **No field type changes.**
   A `number` stays a `number`. A `string` stays a `string`. An object never becomes an array. Null is a valid value for every optional field and the schema says so.

3. **Values may change every day; shape may not.**
   The numbers update on every cron run; that's the product. The *keys* and *types* change on tagged releases of the pipeline, with a minor version bump and a changelog entry.

Enforcement:

- A JSON Schema file (`schemas/countries.schema.json`) is the machine-readable version of this promise.
- The build validates `countries.json` against the schema before publishing. **A schema mismatch fails the build**, which means the live site stays on the last known good dataset.
- The schema file is itself versioned. It has a `$id` field that includes the major version.

Breaking changes ship as **`/api/v2/`** side-by-side with v1 for at least 6 months. During that window both endpoints are generated by the same pipeline pass.

## Consequences

### Positive
- **Downstream consumers are safe.** They can pin to `/api/v1/countries.json` and know it will still be there next year.
- **Client libraries (ADR-0006) stay small.** They can hardcode the field names without a defensive layer.
- **Validation is free.** AJV runs once per build on a 470 KB payload in milliseconds.
- **Schema is documentation.** We don't need a separate OpenAPI spec — the JSON Schema is enough.

### Negative
- **Migrations are expensive.** Shipping `/api/v2/` means running both pipelines in parallel for six months. Acceptable price.
- **Some field names will outlive their usefulness.** `co2.tonnes_per_capita` uses snake_case while most of the JSON is camelCase. It stays snake_case in v1 forever; v2 can rename.

### Neutral
- Per-country files (`countries/{ISO2}.json`) inherit the same promise. They're a view of the same canonical data.

## Alternatives considered

1. **No versioning, "move fast".** Rejected. A dataset that nine upstream parties contribute to and five client libraries wrap is exactly the wrong place to be casual.
2. **Semver on the filename (`countries-v1.json`).** Rejected: the URL is the version. Putting it in both places is noise.
3. **Header-based versioning (`Accept: application/vnd.wihub.v1+json`).** Rejected: GitHub Pages can't negotiate. Static files must live at stable paths.
